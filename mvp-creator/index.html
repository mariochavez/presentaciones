⚠ Anti-pattern Don't open with a tech stack. Starting with "I need a Rails 8.1 app that..." signals a technical task and short-circuits the research and discovery phases.
✦

The golden rule

Lead with the business problem and the people. Defer technical decisions to the discovery questions.

The skill is designed to ask you the right questions — front-loading implementation details bypasses the research phase that makes every document better.

Example trigger

"Help me create an MVP for [business problem].
Target users: [region + persona].
Research: [competitor URLs]."
Research & discovery
Step 1 — Research

Claude searches 2–5 competitors and presents findings before asking any questions. Review carefully before approving.

During the research review

  • Correct any misidentified competitors
  • Flag market gaps Claude may have missed
  • Confirm or adjust the differentiator
  • Validate the feature comparison matrix
Rule Research is the foundation. Even a quick review improves every subsequent document.
Step 2 — Discovery Questions

Answer completely — vague answers produce vague documents. If you don't have an answer, ask Claude to propose options.

  • 01
    App name and meaningIf multilingual — what does it mean?
  • 02
    Core problem in one sentenceThe clearer the better
  • 03
    Primary user personaDemographics, behaviors, pain points
  • 04
    Geographic focus + constraintsAffects language, payments, API availability
  • 05
    Core differentiatorWhat makes this different from what you saw
The five documents
Document Contents What to review
RESEARCH_REPORT.md Competitor analysis, feature matrix, market opportunities, sources Competitor accuracy, gap analysis
MVP_BUSINESS_PLAN.md Vision, user personas, core features, user flows, success metrics Problem statement, feature scope boundaries
BRAND_GUIDE.md Logo SVGs, color palette, typography, Tailwind @theme config Color palette, logo concept, type pairing
TECHNICAL_GUIDE.md Data models, Rails patterns, I18n strategy, architecture decisions Multi-tenancy model, data relationships
CLAUDE.md + setup Project context, MCP config, .claudeignore, custom commands Commands, MCP server configuration
Rule Review and approve each document before requesting the next. Don't batch them — each document informs the one that follows. Don't skip MVP Creator.
Phase 02

Design
& Brand

Two deliverables. One defines how the app speaks. The other defines how it looks. Both inform every SDD spec that follows.

The brand voice document

Brand Guide vs. Brand Voice

The Brand Guide defines visual identity. The Voice document defines how the app speaks.
They are separate deliverables — generate them in that order.

Include in your prompt
Reference brand guideKeeps voice consistent with visual identity
Emotional territory"Economical, deliberate" beats "professional"
The relationship mediatedWho speaks to whom, in what context
Stylistic referenceName a publication or person — not adjectives
Explicit anti-patternsWhat the voice must never sound like
Spanish + English if LATAMBoth languages, examples side by side
✎

The key principle

Give Claude a person to embody, not just adjectives to match.

✓ Specific

"Write it as if a photographer who edits a small literary journal sat down to define how the app speaks. Someone who has opinions about punctuation."

✗ Vague

"Professional, caring about photography."

Shape

4–6 questions, then codebase search.

  • Write references.md
  • Inject standards.md
  • Visuals if needed

Spec

Formal spec from requirements.

  • Goal + User Stories
  • Requirements
  • Visual Design ref
  • Out of Scope

Tasks

4 groups. Each embeds a self-contained Claude Code prompt.

  • DB → Backend
  • Frontend → Integration
  • 2–5 tests/group

Implement

Hand spec folder to Claude Code.

  • Full session or group-by-group
  • Prompts are independent
  • Parallel subagents OK
Recommended feature order Authentication first → Core data model → Core UX flow → Secondary features in priority order.
